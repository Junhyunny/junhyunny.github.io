## 0. 들어가면서

최근 참여했던 프로젝트는 꽤 오랫동안 메인테이닝(maintaining)된 서비스라 기술 부채가 많고 코드 복잡도가 높았다. 특히 프론트엔드 프로젝트의 코드 복잡도가 상당히 높았다. 프론트엔드 컴포넌트 중에서도 새로운 기능을 추가하기 어렵고 버그가 수시로 발생하는 복잡한 컴포넌트가 하나 있었다. 메인 비즈니스와 연관된 컴포넌트였기에 나와 동료는 이 컴포넌트의 리팩토링 우선순위가 높다고 생각했다.

경력이 제법 있는 소프트웨어 엔지니어라면 감각적으로 문제가 있는 코드를 감지할 수 있다. 과거에 비슷한 문제를 겪었다거나, 코드를 읽고 이해하는 데 시간이 걸린다거나, 코드 수정의 영향을 검토하기 어렵다는 등의 경험을 바탕으로 코드 스멜을 포착한다.

하지만 주니어 개발자나 AI는 그렇지 않다. 주니어 개발자는 경험이 적기 때문에 코드만 봤을 때 무엇이 문제인지, 나중에 큰 이자와 함께 돌아올 기술 부채인지, 확장하기 쉬운 구조인지 판단하기 어렵다. 주니어 개발자에게는 어느 쪽이든 똑같이 이해하기 어려운 코드일 가능성이 높다. AI에게도 코드 복잡도는 전혀 의미가 없다. 컨텍스트 윈도우가 허용하는 범위에서 필요한 코드를 모두 읽고 현재 시스템이 어떻게 동작하는지 판단한 후 새로운 기능을 설계하고 구현할 수 있다.

코드가 복잡하니 리팩토링하자는 의견을 어떻게 팀원들에게 설득력 있게 제안할 수 있을까? 어떻게 하면 AI도 시스템의 복잡도를 감지하게 만들 수 있을까? 이번에 함께 일한 동료가 코드 복잡도(code complexity)와 코드 변화 빈도(code churn)에 관해 이야기해 주면서 리팩토링이 시급한 컴포넌트들을 선별했다. 좋은 배움이었다고 생각해 먼저 코드 복잡도에 관한 내용을 글로 정리해 보려 한다.

## 1. 코드 복잡도 (Code Complexity)

코드 복잡도는 IEEE 표준 용어집에 따르면 **"시스템 또는 컴포넌트가 이해하기 어렵고 검증하기 어려운 디자인이나 구현을 가진 정도"**로 정의된다. 이는 단순히 소스 코드의 물리적 길이(라인 수, LOC)만을 뜻하지 않는다. 코드 구조의 난해함, 연산자와 피연산자의 수, 제어 흐름(if-else)의 수, 모듈이나 객체 간의 결합도 등 다양한 요인이 얽혀 복잡도를 결정한다.

소프트웨어 공학의 석학 프레드 브룩스(Fred Brooks)는 코드 복잡성을 그 근원에 따라 두 가지로 구분했다.

- 본질적 복잡성 (Essential Complexity)
  - 해결하고자 하는 문제 영역(도메인) 자체에 내재된 고유한 어려움이다. 
  - 비즈니스 요구사항이나 알고리즘 자체의 수학적 난이도를 의미한다. 
  - 원래의 요구사항을 변경하지 않는 한 이 복잡성은 절대 제거할 수 없다.
- 우발적 복잡성 (Accidental Complexity)
  - 개발 과정에서 선택한 도구, 서툰 설계, 오버엔지니어링 혹은 비효율적인 분리 등으로 인해 인위적으로 부가된 어려움이다.
  - 이는 문제 본질과는 무관한 복잡성이므로, 리팩토링이나 단순화를 통해 소프트웨어 엔지니어가 적극적으로 줄이고 피할 수 있는 대상이 된다.

본질적 복잡성과 우발적 복잡성이 소프트웨어가 복잡해지는 원인을 기준으로 한 구분이라면, 다음 지표들은 구현된 코드의 복잡도를 측정하는 관점에서 구분한 것이다.

- 순환 복잡도 
  - 조건문과 반복문으로 만들어지는 선형 독립 실행 경로의 수를 측정한다. 값이 높을수록 검사해야 할 기본 경로가 많아진다.
- 인지 복잡도 
  - 개발자가 코드를 읽고 이해할 때 느끼는 상대적인 부담을 측정한다. 특히 깊은 중첩이나 복잡한 흐름 전환에 높은 점수를 부여한다.
- NPath 복잡도 
  - 함수 안에서 반복을 제외하고 실행할 수 있는 경로 조합의 수를 나타낸다. 독립적인 분기가 누적되면 값이 빠르게 증가할 수 있다.
- Halstead 복잡도 
  - 연산자와 피연산자의 종류 및 사용 횟수를 바탕으로 코드의 크기, 난도, 구현 노력을 추정한다.

이번 글에서는 순환 복잡도와 인지 복잡도를 구체적으로 정리했다.

## 2. 순환 복잡도 (Cyclomatic Complexity)

먼저 순환 복잡도를 살펴보자. 1976년 토마스 맥케이브(Thomas J. McCabe)가 제안한 대표적인 구조적 복잡도 측정 지표다. 소스 코드 내에 존재하는 선형적으로 독립적인 실행 경로의 개수를 계산한다. 제어 흐름 그래프(control flow graph)상에서 결정 지점(조건문, 루프 등)이 하나 추가될 때마다 복잡도가 높아진다.

코드를 제어 흐름 그래프(Control Flow Graph)로 표현한 뒤 다음 공식을 사용한다.

- M: 순환 복잡도
- E: 그래프의 간선 수
- N: 그래프의 노드 수
- P: 연결된 컴포넌트 수

```
M = E - N + 2P
```

제어 흐름 그래프는 무엇일까? 간단한 예제 코드와 함께 살펴보자. 다음 메서드는 18세 이상이면 "adult", 18세 미만이면 "minor"라는 결괏값을 반환한다.
 
```java
String checkAge(int age) {
    if (age >= 18) {
        return "adult";
    }
    return "minor";
}
```

위 그래프를 바탕으로 순환 복잡도를 계산하면 "2"라는 값을 얻을 수 있다.

- 노드 N = 5
- 간선 E = 5
- 연결 요소 P = 1

```
M = E - N + 2P
  = 5 - 5 + 2
  = 2
```

분기점의 수를 세어 순환 복잡도를 간단히 계산하는 방법도 있다.

- 위 예제 코드는 if 구문이 한 개 존재하므로 순환 복잡도는 "2"로 동일하다.

```
순환 복잡도 = 분기점 수 + 1
```

대표적인 분기점은 다음과 같다. switch, &&, `||`, 예외 처리 등을 세는 세부 규칙은 분석 도구와 언어에 따라 달라질 수 있다. 따라서 프로젝트에서 같은 도구와 설정을 일관되게 사용하는 것이 중요하다.

- if, else if
- for, while, do-while
- case
- catch
- 삼항 연산자
- 단락 평가가 발생하는 &&, `||`

순환 복잡도를 알면 함수 안에 분기 경로가 얼마나 많은지, 어떤 함수에 이해하거나 변경하기 어려운 복잡도가 집중되어 있는지, 코드를 충분히 검증하려면 최소 몇 개의 독립적인 테스트 경로가 필요한지 파악할 수 있다. 테스트 경로에 관한 내용을 조금 더 구체적으로 살펴보자. 순환 복잡도는 프로그램의 제어 흐름 그래프를 기반으로 선형적으로 독립적인 실행 경로(Linearly Independent Paths)의 개수를 정량화한 수치로 볼 수 있다.

토마스 맥케이브(McCabe)가 제안한 '기반 경로 테스팅' 전략에 따르면, **프로그램 내의 모든 독립적인 실행 경로를 최소 한 번씩 검증하는 데 필요한 테스트 케이스의 수는 순환 복잡도 수치와 동일하다.** 순환 복잡도는 코드를 완전히 테스트하기 위해 실행해야 하는 **최소한의 테스트 횟수(하한선)**를 제공한다. 만약 어떤 함수나 모듈에 대해 설계된 테스트 케이스의 개수가 그 코드의 순환 복잡도 값보다 작다면, 모든 독립 경로를 온전히 검증할 수 없음을 의미한다.

예를 들어, 위에서 살펴본 checkAge 메서드의 순환 복잡도 점수가 2로 측정됐으므로 모든 경로를 충분히 커버하려면 서로 다른 테스트 케이스를 최소 2개 설계해야 한다.

```java
assertEquals("adult", checkAge(18));
assertEquals("minor", checkAge(17));
```

순환 복잡도 점수가 요구하는 최소한의 테스트 수를 충족했다고 해서 올바른 테스트가 수행되었음을 완전히 보장하는 것은 아니며, 실제 로직의 완벽한 안전성을 검증하기 위해 실무에서 요구되는 총 테스트 케이스의 수는 순환 복잡도 값보다 훨씬 더 많아질 수 있다.

미국 국토안보부(DHS)의 발표 자료와 소프트웨어 공학계에서 통용되는 순환 복잡도 구간별 해석은 다음과 같다. 순환 복잡도가 높을수록 복잡도를 낮추기 위한 리팩토링이 필요한 상태다. 최근 참여했던 프로젝트에서는 순환 복잡도가 72로 매우 높았던 컴포넌트를 20 미만으로 낮추는 리팩토링을 수행했다.

| 순환 복잡도 | 복잡도 / 위험도 | 해석 |
|---|---|---|
| 1 ~ 10 | 낮은 복잡도 / 저위험 | 구조가 단순하며 오류 발생 위험이 적은 단계이다. 테스트와 유지보수가 용이한 안정적인 상태로 판단한다. |
| 11 ~ 20 | 보통 복잡도 / 중위험 | 코드가 다소 복잡해지고 위험도가 중간 수준인 단계이다. 이 단계부터는 추가적인 테스팅 노력이 요구된다. |
| 21 ~ 50 | 높은 복잡도 / 고위험 | 코드가 매우 복잡하고 위험도가 높은 단계이다. 많은 결정 지점으로 인해 결함이 발생할 위험이 증가하며 테스트 검증이 매우 까다로워진다. 참고로 .NET 정적 분석이나 Visual Studio 가이드라인인 CA1502 규칙 등은 복잡도가 25에 도달할 때 경고를 실행하여 과도한 복잡성을 방지한다. |
| 50 초과 | 극도로 높은 복잡도 / 극도의 고위험 | 불안정하며 위험이 아주 높은 단계이다. 사실상 "테스트 불가능(untestable)" 상태로 분류되며, 시스템 품질 향상을 위해 즉각적인 리팩토링이 강력히 요구된다. |

언어별로 순환 복잡도를 측정할 수 있는 도구가 있다. 자주 사용하는 언어를 지원하는 도구 가운데 무료로 사용할 수 있고 CLI에서 실행 가능한 도구는 다음과 같다.

| 언어 | 도구 | 검사 방식 | 특징 |
|---|---|---|---|
| Java | Checkstyle | 임계값 초과 탐지 | `CyclomaticComplexity` 규칙으로 허용 가능한 복잡도를 초과하는 메서드를 탐지하며 Gradle·Maven과 연동할 수 있다. |
| Java | PMD | 임계값 초과 탐지 | 순환 복잡도 규칙을 통해 복잡한 메서드를 탐지하고 XML·HTML·SARIF 등의 보고서를 생성할 수 있다. |
| Kotlin | detekt | 임계값 탐지·복잡도 보고서 | `CyclomaticComplexMethod` 규칙으로 Kotlin 함수의 순환 복잡도를 검사한다. Kotlin 스코프 함수와 `when`, Elvis 연산자 등의 문법도 고려한다. |
| JavaScript / TypeScript | ESLint | 임계값 초과 탐지 | 내장 `complexity` 규칙을 사용해 함수의 순환 복잡도가 설정값을 초과하면 경고하거나 오류로 처리한다. |
| Python | PyLint | 분기 수 초과 탐지 | `too-many-branches` 규칙으로 조건 분기가 지나치게 많은 함수나 메서드를 탐지한다. 엄밀한 McCabe 복잡도 수치보다는 구조적 대리 지표에 가깝다. |

## 3. 인지 복잡도 (Cognitive Complexity)

**인지 복잡도**는 정적 분석 플랫폼인 소나소스(SonarSource)가 기존 순환 복잡도의 한계를 보완하고 코드의 이해 난이도를 정량화하기 위해 개발한 소프트웨어 품질 메트릭이다. 순환 복잡도가 제어 흐름 그래프를 기반으로 소스 코드의 기계적 구조와 실행 경로의 수를 계산하는 것과 달리, 인지 복잡도는 개발자가 코드를 읽고 직관적으로 이해하는 데 필요한 인지적 부담을 정량화하는 데 초점을 맞췄다.

인지 복잡도 점수는 0점에서 시작하며, 사람이 코드를 위에서 아래로 순차적으로 읽어 나갈 때 인지적 흐름을 방해하는 특정 패턴이 나타날 때마다 누적된다.

- 가독성을 높이는 축약 구조 무시
  - 단순한 메서드 호출처럼 가독성을 높이기 위해 하나로 압축된 선언 구조는 복잡도를 증가시키지 않는다. 단, 재귀 호출은 가산한다.
  - try·finally 구문도 무시한다. 
- 선형 흐름 중단 시 가산
  - 코드의 선형 흐름을 방해하는 제어 구조(if, else, for, while, catch, goto, 재귀 등)를 만날 때마다 점수가 1점씩 추가된다.
- 중첩 페널티(nesting penalty)
  - 루프나 조건문 내부에 또 다른 루프나 조건문이 들어가는 **중첩 구조(Nesting)**가 깊어질수록 해당 제어 구조를 이해하는 데 훨씬 큰 정신적 노력이 요구된다.
  - 따라서 중첩이 깊어질수록 가중치 점수가 현재 중첩 깊이만큼 추가된다.
- 논리 연산자 시퀀스 가산
  - 조건문 내부에 &&나 `||`와 같은 복잡한 논리 연산자 체인이 연속해서 나타나면 인지적 과부하를 고려하여 점수가 추가된다.

간단한 예제 코드를 살펴보자. 아래처럼 일반적인 순차 실행과 메서드 호출은 인지 복잡도가 0점이다. 흐름을 방해하지 않으므로 가산되지 않는다. 

```java
void process(Order order) {
    validate(order);
    save(order);
    notify(order);
}
```

아래 코드처럼 try-finally 구문은 무시한다. 아래 코드는 인지 복잡도가 0점이다.

```java
void save() {
    try {
        repository.save();
    } finally {
        repository.close();
    }
}
```

예외 분기를 만드는 catch 블록은 가산한다. 아래 코드는 인지 복잡도가 2점이다.

```java
void load() {
    try {
        repository.load();
    } catch (IOException e) { // +1
        handleIoError(e);
    } catch (SecurityException e) { // +1
        handleSecurityError(e);
    } finally { // +0
        repository.close();
    }
}
```

지금부터 가산되는 예를 살펴보자. 단순한 if 구문은 복잡도 점수를 높인다. 아래 코드는 인지 복잡도가 1점이다.

```java
void process(boolean valid) {
    if (valid) { // +1
        execute();
    }
}
```

else if와 else도 각각 흐름 전환으로 계산하기 때문에 인지 복잡도를 높인다. 아래 코드는 인지 복잡도가 3점이다.

```java
String classify(int score) {
    if (score >= 90) { // +1
        return "A";
    } else if (score >= 80) { // +1
        return "B";
    } else { // +1
        return "C";
    }
}
```

중첩이 깊어질수록 각 제어 구조에 현재 깊이가 추가된다. 아래 코드는 인지 복잡도가 6점이다.

```java
void process(boolean active, List<Item> items) {
    if (active) { // +1
        for (Item item : items) { // +2: 기본 1 + 중첩 1
            if (item.isValid()) { // +3: 기본 1 + 중첩 2
                execute(item);
            }
        }
    }
}
```

switch-case 구문은 전체에 1점만 부여하고 각각의 case에는 점수를 추가하지 않는다. 여러 조건식을 사용할 수 있는 if-else나 if보다 비교적 이해하기 쉬운 구조로 보기 때문이다. 아래 코드는 인지 복잡도가 1점이다.

```java
String describe(Status status) {
    switch (status) { // +1
        case NEW:
            return "new";
        case PAID:
            return "paid";
        default:
            return "done";
    }
}
```

논리 연산자는 연산자의 개수가 아니라 동일한 연산자가 연속되는 시퀀스를 계산한다. 동일한 연산자가 이어지면 하나의 시퀀스로 본다. 아래 코드는 인지 복잡도가 1점이다.

```java
if (a && b && c) { // if +1, && 시퀀스 +1
    execute();
}
```

논리 연산자가 바뀌는 경우 인지 복잡도가 높아진다. 아래 코드는 인지 복잡도가 4점이다.

```java
if (a && b || c && d) { // if +1, && 시퀀스 +1, || 시퀀스 +1, 새로운 && +1
    execute();
}
```

삼항 연산자는 조건 흐름이 있으므로 무시되지 않는다. 다만 if-else와 달리 else 부분에 별도 점수를 부여하지 않는 할인된 방식으로 계산한다. 아래 코드는 인지 복잡도가 1점이다.

```java
String result = valid ? "success" : "failure"; // +1
```

일반 메서드 호출은 무시하지만, 직접 재귀 또는 간접 재귀 사이클에 참여하는 메서드에는 점수를 추가한다. 아래 코드는 인지 복잡도가 2점이다.

```java
int factorial(int number) {
    if (number <= 1) { // +1
        return 1;
    }
    return number * factorial(number - 1); // 재귀 +1
}
```

람다 내부에서 중첩이 발생해도 인지 복잡도 점수가 추가된다. 람다 자체에는 점수를 추가하지 않지만, 람다 내부의 제어 구조에는 중첩 페널티가 적용된다. 아래 코드는 인지 복잡도가 2점이다.

```java
Runnable task(boolean active) {
    return () -> { // +0, 중첩 깊이는 증가
        if (active) { // +2: 기본 1 + 중첩 1
            execute();
        }
    };
}
```

흐름 중단 구문은 일반적으로 별도 가산하지 않는다.

- 일반 return
- 일반 throw
- 반복문 자체를 빠져나가는 일반 break
- 현재 반복을 넘기는 일반 continue

하지만 라벨을 사용해 멀리 이동하는 흐름은 가산된다. 자바나 자바스크립트에서 본 적은 없지만, 대학 시절 C 언어 문법에서 본 기억이 있다. 라벨이 있는 continue 구문은 현재 위치에서 멀리 떨어진 외부 흐름으로 이동하므로 별도의 인지 부담으로 계산한다. 아래 코드는 인지 복잡도가 7점이다.

```java
OUTER:
for (Item item : items) { // +1
    for (Option option : options) { // +2
        if (matches(item, option)) { // +3
            continue OUTER; // +1
        }
    }
}
```

인지 복잡도 점수에 따라 어떤 조치를 취하는 것이 좋을까? 아래 표는 인지 복잡도 점수를 어떻게 해석하면 좋을지 정리한 것이다.

| 인지 복잡도 | 복잡도 수준 | 해석 |
|---:|---|---|
| `0` | 매우 낮음 | 제어 흐름 변화가 없는 순차적인 코드로, 이해 부담이 거의 없다. |
| `1~5` | 낮음 | 조건이나 반복이 소수 존재하지만 전체 흐름을 쉽게 파악할 수 있다. |
| `6~10` | 보통 | 여러 분기나 일부 중첩이 존재한다. 일반적으로 관리 가능하지만 변경 시 전체 흐름을 확인할 필요가 있다. |
| `11~15` | 다소 높음 | 분기와 중첩이 누적되어 코드를 한 번에 이해하기 어려워질 수 있다. 리팩터링 가능성을 검토하는 것이 좋다. |
| `16~25` | 높음 | 제어 흐름을 추적하는 데 상당한 인지 부담이 발생한다. 함수 분리, 중첩 완화, 조건 단순화가 필요한지 적극적으로 검토한다. |
| `26 이상` | 매우 높음 | 코드의 이해·검토·변경이 어렵고 실수 가능성이 커질 수 있다. 우선적인 분석과 리팩터링이 권장된다. |


순환 복잡도와 마찬가지로 언어별 인지 복잡도를 측정할 수 있는 도구가 있다. 자주 사용하는 언어를 지원하면서 무료로 사용할 수 있고 CLI에서 실행 가능한 도구는 다음과 같다.

| 언어 | 도구 | 인지 복잡도 검사 기능 |
|---|---|---|
| Java | PMD | `CognitiveComplexity` 규칙으로 메서드별 임계값 초과 탐지 |
| Kotlin | detekt | `CognitiveComplexMethod` 규칙으로 함수별 복잡도 검사 |
| Python | complexipy | 함수·메서드의 인지 복잡도 수치 계산 및 임계값 검사 |
| JavaScript | ESLint + `eslint-plugin-sonarjs` | `sonarjs/cognitive-complexity` 규칙 사용 |
| TypeScript | ESLint + `eslint-plugin-sonarjs` | 동일한 규칙 사용. TypeScript 파서 설정 필요 |

## CLOSING

다중 언어를 지원하며 CI/CD 파이프라인과 연동할 수 있는 플랫폼도 존재한다. SonarQube 플랫폼은 순환 복잡도와 인지 복잡도를 모두 지원한다.

| 플랫폼 | 형태 | 특징 |
|---|---|---|
| SonarQube / SonarCloud / SonarLint | 서버, 클라우드, IDE 플러그인 | 여러 언어와 프레임워크를 정적 분석하고 순환 복잡도와 **인지 복잡도(Cognitive Complexity)**를 리포트한다. 품질 게이트(Quality Gate)를 CI/CD 파이프라인에 연동해 기준을 위반한 코드의 병합을 제한할 수 있다. |
| Codacy | 클라우드 플랫폼 | 리포지토리를 자동으로 스캔해 순환 복잡도, 코드 중복, 기술 부채의 추세를 대시보드로 시각화한다. |
| CodeScene | 코드 분석 플랫폼 | 코드 변경 빈도와 순환 복잡도를 결합해 핫스팟을 분석하고, 중첩된 조건문의 인지적 부담을 시각적인 Bumpy Road 패턴으로 탐지한다. |

순환 복잡도와 인지 복잡도를 감지할 수 있는 언어별 CLI 도구를 [AI 코딩 에이전트 하네스(harness)][harness-engineering-link]에 추가해 볼 생각이다. AI가 구현을 마치고 커밋하기 전에 코드의 복잡도를 판단하고 리팩토링할 수 있도록 토이 프로젝트 훅(hook)에 적용할 계획이다. 실험을 마치면 관련 내용을 블로그에 업데이트할 예정이다.
